# 智慧便民服务平台重构协作规则

## 1. 项目目标

本项目以现有便捷生活类 HarmonyOS 源码为起点进行重构，不要求与原源码的页面、数据和业务流程完全一致。

重构目标：

- 保留 Stage 模型、模块化结构和有价值的系统能力示例。
- 重新设计页面信息架构、数据层、交互闭环、错误状态和安全边界。
- 六名成员各自负责一个完整、可运行、可截图、可撰写报告的区域。
- 最终成果必须能在全组统一的 DevEco Studio/SDK 环境中编译、安装和运行。

不要把静态页面、空点击事件或“功能还在开发中”Toast 当作已完成功能；不要把 Mock 数据描述成真实接口。

## 2. 六人职责边界

| 成员 | 负责区域 | 主要目录/文件 | 一句话职责 |
|---|---|---|---|
| A | 整体框架与项目整合 | `entry/`、`features/RouterModule/`、`common/common/`、项目配置 | 让整个项目可以组合、跳转、编译和运行 |
| B | 首页与智慧工具 | `features/home/.../HomePage.ets`、扫码/二维码/证件识别、`features/home/Index.ets` | 完成首页、服务入口和智慧识别工具 |
| C | 民生服务 | `features/home/.../SocialSecurityPage.ets`、`ChequeSheetPage.ets`、`RentingHousePage.ets`、`ResidencePermitPage.ets` 及相关模型 | 完成社保、账单、租房、居住证等业务详情 |
| D | 办事大厅与网络 | `features/office/`、`common/network/` | 完成办事分类、搜索、详情和统一网络访问 |
| E | 互动与 12345 | `features/message/` | 完成诉求提交、记录、状态查询和互动信息 |
| F | 我的与账户 | `features/mine/` | 完成登录、注册、个人中心、状态持久化和安全处理 |

职责边界说明：

- B 负责首页上的“社保、账单、租房”等入口，C 负责点击入口后的详细页面。
- `features/home/Index.ets` 和首页路由注册由 B 统一修改；C 把新增页面名称和参数交给 B 接入。
- `common/network/` 由 D 统一修改；其他成员只调用 D 暴露的类型化业务接口。
- `features/mine/` 的内部存储由 F 管理；其他模块只能调用登录状态和跳转登录等公开接口。
- A 管理公共路由、项目配置和公共资源，不代替 B—F 完成各自业务功能。
- 除非当前任务明确需要跨模块修改，否则不要越过上述文件所有权。
- 以现有源码做兼容式完善：已经被其他模块使用的包名、`Index.ets` 导出、路由常量和调用方式默认保留；新增接口优先采用“旧代码可继续运行、成员可逐步迁移”的方式。确需破坏性调整时，由 A 先列出全部调用方、迁移步骤和统一合并时间。

## 3. 必须遵守的任务顺序

### 阶段 0：统一环境并验证基线

负责人：A 主导，全员参加。此阶段完成前不要大规模改代码。

1. A 在基准电脑上同步工程并执行一次干净构建。
2. 记录 DevEco Studio、HarmonyOS SDK、Hvigor、ohpm 和 Node.js 的实际版本。
3. 全员统一使用同一套版本；任何人不得自行升级项目 SDK。
4. 如果老师没有锁定 SDK，可由全组评估稳定 Release SDK；API 26 Beta 不作为默认提交基线。
5. 记录当前编译错误、警告和可运行页面，形成基线清单。

通过条件：基线构建结果已经记录。若未实际构建，只能写“未验证”，不能宣称工程可运行。

### 阶段 1：冻结范围和接口

负责人：A 汇总，B—F 提供本模块需求。

1. 六人分别列出本模块的 Must 功能，不在此阶段追求动画和可选增强。
2. A 冻结主 Tab、页面名称、路由参数和返回结果。
3. B 与 C 冻结首页入口 ID 和民生页面路由。
4. D 冻结 `NetworkService` 的公开方法、返回模型、错误模型和 Mock 开关。
5. F 冻结 `SessionService` 的登录状态和跳转登录接口。
6. E 冻结诉求记录的数据表字段和状态枚举。

通过条件：每个模块都有入口、输入、输出、数据方案、负责人和验收路径。

### 阶段 2：公共基础与模块骨架并行开发

此阶段可以并行，不需要六个人排队：

- A：完成应用壳、四个 Tab、Navigation/NavPathStack、公共资源和错误状态组件。
- B：首页先完成入口模型和页面骨架，使用约定的路由，不等待 C 的最终页面。
- C：完成民生数据模型、Mock Repository 和各详情页骨架。
- D：优先完成 NetworkService、统一结果模型、Mock 回退和 ArkWeb 公共页。
- E：完成诉求表单模型、relationalStore 表结构和互动页骨架。
- F：优先完成 SessionService、登录状态持久化和账户页骨架。

依赖原则：D 和 F 的公开接口要尽早稳定；其他成员可先使用同接口的 Mock 实现，不能因为等待接口而停工。

通过条件：六个负责区域都能从主框架或独立预览进入，且不存在编译阻断错误。

### 阶段 3：六个业务模块并行完成 Must 流程

- A：完善路由返回、安全区、公共加载/空/错状态，并持续集成。
- B：完成首页、搜索、扫码、二维码、证件识别及异常反馈。
- C：至少完成三条民生业务流程，例如社保、账单、租房，并加入筛选或详情。
- D：完成办事分类、搜索、详情、网络/Mock 请求、错误和重试。
- E：完成诉求提交、保存、列表、详情和状态筛选闭环。
- F：完成输入校验、模拟登录、个人中心、状态恢复和退出登录。

通过条件：每名成员至少有一条从入口到结果的完整演示路径；不能只完成静态 UI。

### 阶段 4：成员自测和证据收集

每名成员先自测自己的模块，再交给 A 集成：

1. 测试正常路径、空数据、错误输入或失败路径。
2. 为关键纯逻辑补充有业务意义的单元测试；不要只保留模板 `abc` 测试。
3. 在模拟器或真机上实际运行。
4. 保存至少 3 张带状态栏的运行截图。
5. 保存至少 2 个真实问题的原始报错、原因和解决过程。
6. 标注已知限制和未验证设备能力。

通过条件：模块代码、测试路径、截图、问题记录和已知限制同时提交。

### 阶段 5：按依赖顺序合并与联调

由 A 在基线分支上按以下顺序集成：

1. A 的应用壳、公共路由和公共模型。
2. D 的网络服务和办事模块。
3. F 的会话服务和账户模块。
4. C 的民生详情页面和数据层。
5. E 的互动与诉求模块。
6. B 的首页和所有业务入口，作为业务导航收口。
7. A 执行全局回归、修复跨模块问题并生成候选提交版本。

这样安排的原因：先合并被其他模块依赖的网络与会话能力，再合并具体业务，最后用首页统一连接所有入口。

不要等到最后一次性合并；至少在模块骨架完成、Must 完成和候选版本三个节点做增量合并。

### 阶段 6：报告与最终提交

1. 截图完成后再冻结候选版本，报告中的代码和截图必须与提交工程一致。
2. A 撰写项目概述、架构、模块分工和最终集成部分。
3. B—F 分别撰写自己的模块章节、数据方案、问题解决和个人总结。
4. A 在另一台电脑或干净环境完成最终构建、安装和核心流程回归。
5. 打包完整工程，导出 DOCX/PDF，准备 5 分钟展示。

## 4. 开发与架构规则

- 使用 HarmonyOS Stage 模型，不引入 FA 模型写法。
- 项目 SDK 以阶段 0 冻结结果为准；未经 A 或用户确认，不修改 SDK/Hvigor/ohpm 版本。
- 新页面优先使用 `Navigation`、`NavDestination` 和 `NavPathStack`；不要新增旧式 `router.push()`。
- 更新代码优先使用 `@kit.*` 导入和 `UIContext` 获取上下文/Toast；不要为了“统一格式”一次性改写无关文件。
- V1/V2 状态管理按冻结的目标 SDK 和模块边界选择；不要在同一数据链路随意混用。
- 页面不得直接创建重复的网络实现；通过 D 的公开服务访问数据。
- 所有数据来源必须如实标注为硬编码、Mock、真实接口、Preferences 或 relationalStore。
- 真实接口必须有文档；接口不可用时使用明确标注的 Mock 回退，不伪造成功结果。
- 列表、网络和数据库页面至少考虑加载、空数据、错误、成功四种状态中的三种。
- 用户授权权限必须同时在 `module.json5` 声明并在运行时申请；需要 `reason`/`usedScene` 时必须补齐。
- 不在代码中硬编码密钥、IV、密码或 Token，不在 HiLog 中输出手机号、证件信息、密码和 Token。
- 敏感短数据使用 Asset Store；轻量设置使用 Preferences；结构化记录使用 relationalStore。
- 注册事件监听后必须在合适生命周期解除；图片、ResultSet、HTTP 请求等资源要及时释放。

## 5. 跨模块接口规则

- 路由名称、参数和返回值使用明确类型，不散落魔法字符串。
- 跨模块只从各模块公开的 `Index.ets` 导入；不要新增指向其他成员 `src/main/ets/...` 的深层导入。历史深层导入暂时保留，等对应模块公开导出后再由 A 统一迁移。
- 公共错误模型至少包含错误码、用户提示和是否允许重试。
- D 的网络层只暴露业务方法，不向业务页面暴露底层 `http` 实例。
- F 的会话层只暴露登录状态、显示名称和跳转登录方法，不向其他模块暴露密码或 Token。
- B/C 共用的服务入口模型至少包含 `id`、标题、图标、路由和是否需要登录。
- 改动共享接口前先搜索所有调用方，并同步修改测试和调用代码。

## 6. 验证与完成定义

一个模块只有同时满足以下条件才算完成：

- Must 功能形成“入口—操作—结果”的完整流程。
- 在冻结的开发环境中通过实际构建；未构建不得标记完成。
- 在模拟器或真机上完成至少一条核心路径验证。
- 不存在用空点击或“功能还在开发中”代替的 Must 功能。
- 数据来源与报告描述一致。
- 敏感信息没有明文存储或日志泄露。
- 有真实测试路径、至少 3 张运行截图和至少 2 个问题记录。
- 合并后不破坏其他成员模块。

运行构建前先检查仓库现有脚本和本机工具帮助，不猜测或编造 DevEco CLI/Hvigor 命令。设备能力无法验证时，明确记录“未验证”和原因。

## 7. 报告材料要求

每名成员负责自己的代码和报告证据：

- 模块详细实现不少于 1500 字（含代码）。
- 选择 3—5 段关键代码进行分析。
- 至少 3 张带模拟器状态栏的运行截图。
- 至少 2 个真实问题，包含原始报错、原因、解决步骤和结果。
- 明确说明数据方案和使用的新知识点/新组件。

全组最低材料量：

- 6 个成员模块章节，总计不少于 9000 字。
- 至少 18 张运行截图。
- 至少 12 个真实问题。
- 共 18—30 段关键代码分析。
- 最终提交 DOCX/PDF、报告中的源代码文本和完整工程压缩包。

报告命名按任务书执行：`组名_生产实习报告_姓名`。

## 8. 修改前后检查清单

开始修改前：

- 确认当前任务属于 A—F 中哪个负责区域。
- 阅读目标模块的 `module.json5`、`oh-package.json5` 和公开 `Index.ets`。
- 检查是否会修改共享接口或其他成员拥有的文件。
- 明确数据来源和可验证的完成条件。

提交修改前：

- 检查编译错误、废弃 API、未使用导入和资源路径。
- 验证正常、空、错或取消路径。
- 检查日志、密钥、Token 和个人信息。
- 运行与改动风险相匹配的测试和设备验证。
- 记录仍未验证的内容，不把部分完成写成全部完成。
